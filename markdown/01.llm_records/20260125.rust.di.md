# Dependency Injection Framework in Rust

by OpenAI GPT5.2

Below is a **short-list of three Rust DI projects that are the best fit for embedded / industrial applications**, selected explicitly on **maturity, predictability, runtime characteristics, and suitability for long-lived systems** (PLC-like controllers, gateways, field devices, edge nodes). I also explain *why some popular DI styles are excluded* for this domain.

---

## Executive Summary (Top 3)

| Rank  | Project                   | Why it fits embedded / industrial                                                     |
| ----- | ------------------------- | ------------------------------------------------------------------------------------- |
| **1** | **`dependency_injector`** | Most production-oriented runtime DI; predictable behavior, thread-safe, minimal magic |
| **2** | **`more-di`**             | Balanced design; flexible but not overly complex; supports async and modular systems  |
| **3** | **`je_di`**               | Compile-time DI; strongest determinism and lowest runtime overhead                    |

---

## 1. `dependency_injector` — **Best overall choice**

### Maturity

* Moderate but **clearly production-oriented**
* Well-documented container semantics
* Designed with concurrency and performance as first-class concerns

### Why it is suitable for industrial embedded systems

* **No reflection**
* **Explicit registration and resolution**
* **Predictable lifetimes** (singleton / scoped / transient)
* Designed to work under **multi-threaded, long-running conditions**
* No reliance on heavy procedural macro “magic”

### Typical embedded / industrial use cases

* Edge gateways (industrial IoT)
* Controller software running on embedded Linux
* Modular systems with replaceable drivers (CAN, Modbus, OPC-UA, GPIO)
* Service-oriented device firmware where components must be hot-swappable or mockable

### Why it ranks #1

It aligns best with **Celix-like thinking**:

* Service registry
* Explicit lifecycle management
* Runtime composition
* Clear ownership and threading guarantees

## 2. `more-di` — **Best balance of flexibility and restraint**

### Maturity

* Moderate, stable evolution
* Multiple releases and active documentation
* Used as a *foundation* rather than a “framework”

### Why it works for embedded contexts

* Supports **async** (important for IO-heavy field devices)
* Allows **explicit container wiring**
* Does not force a web-framework or application-level abstraction model
* Acceptable runtime cost when used conservatively

### Typical use cases

* Embedded Linux systems with async runtimes
* Industrial protocol stacks (MQTT, OPC-UA, fieldbus bridges)
* Systems where compile-time DI is too rigid, but full runtime frameworks are too heavy

### Trade-off

* Slightly more flexible (and thus less rigidly deterministic) than compile-time DI
* Still appropriate when **configuration may vary per deployment**

---

## 3. `je_di` — **Best for hard real-time and safety-oriented designs**

### Maturity

* Early, but conceptually solid
* Narrow scope, which is a *strength* in embedded systems

### Why it fits embedded / industry

* **Compile-time DI** → zero runtime container cost
* Strong determinism
* No heap-heavy runtime resolution
* Failures surface at compile time, not in the field

### Ideal use cases

* Bare-metal or RTOS-like environments
* Safety-critical controllers
* Systems with **static topology** (no dynamic module loading)
* Firmware with strict certification or validation requirements

### Limitation (important)

* Less flexible
* Poor fit if:

  * Services must be added/removed dynamically
  * Configuration changes frequently per deployment

---

## Why other DI crates are *not* recommended for industrial embedded use

### `ferrous-di`

* Too **enterprise-style**
* AOP, interception, and advanced features increase:

  * Binary size
  * Cognitive load
  * Runtime unpredictability
* Better suited for **server-side services**, not field devices

### `rustyinject`, `chassis`, similar

* Experimental
* Low ecosystem traction
* Risky for long-lifecycle industrial products (10–20 years)

### Legacy projects (e.g., Shaku)

* Largely unmaintained
* Risky dependency profile

## Practical Recommendation Pattern

For **industrial embedded Rust**, I recommend:

* **Dynamic, service-oriented devices** → `dependency_injector`
* **Configurable edge systems** → `more-di`
* **Hard real-time / safety systems** → `je_di`
